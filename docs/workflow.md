# 工作流

## 1. 输入来源

支持两种主要输入：

1. 对话型输入
   - 我们围绕一个 AI 话题展开讨论
   - 结束后，我整理结论并写成文章
2. 粘贴型输入
   - 你把外部聊天记录、随笔、提纲或半成品文章直接贴过来
   - 我负责重组、提炼、改写成公众号文章

## 2. 标准产出链路

1. 先明确文章主题
2. 提炼核心观点
3. 确定目标读者和阅读收益
4. 按模板组织成文
5. 检查语言是否通俗、案例是否具体、标题是否清晰
6. 文章图片放入 `assets/`
7. 生成公众号横版封面，比例为 `2.35:1`
8. 复制 3 张需上传到 `mdnice` 的图片到本地 `~/Downloads`，按 `1`、`2`、`3` 命名
9. 在 `mdnice` 中手动上传图片并完成发布
10. 归档成稿，并更新格式记忆

## 3. 每次写作默认检查项

- 标题是否足够具体，而不是空泛
- 开头是否快速交代问题和收益
- 正文是否有清晰的小标题
- 是否尽量避免抽象套话
- 是否给出可操作建议或判断框架
- 结尾是否有一句能收住全文的话

## 4. 迭代规则

以下内容后续都可以被修订：

- 文章固定结构
- 标题风格
- 结尾样式
- 是否保留金句、小结、行动建议
- 是否增加配图建议、摘要、封面文案

最终以 `docs/article-format.md` 为准。
# 提交流程

- 写作规范、格式记忆、流程说明这类基础规则的修改，单独提交一个 commit
- 这类 commit 的 message 直接写本次修改的主要内容，不要和当天文章混在一起
- 每天文章的正文、配图、相关素材单独提交一个 commit
- 文章标题默认使用 `YYYY-MM-DD: 实际标题`
- 文章 commit 的 message 使用 `YYYY.MM.DD: 实际标题`
- 如果同一天既修改了规范又产出了文章，默认拆成两个 commit
