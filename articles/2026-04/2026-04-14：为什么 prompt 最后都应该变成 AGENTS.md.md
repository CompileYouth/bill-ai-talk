> TL;DR
>
> Prompt 适合交代这一次怎么做，不适合把所有长期规则都塞进去。规则一多，Prompt 只会越写越长，越改越乱。更好的做法，是把真正要长期稳定下来的东西写进 AGENTS.md，让它从一段临时提示词，变成一个可以一直继承、一直迭代的系统。

AI 时代写文章，多少都离不开 AI 的帮助。可只要想让每次生成出来的文章都更接近自己的预期，最基础的一步，就是先维护一段稳定的 Prompt。问题是，写着写着就会发现，一段 Prompt 很快就不够用了。最开始你可能只想交代这篇文章写什么，后来又想把语言风格、标题要求、TL;DR、封面词、配图规则、Reviewer 的检查标准一起塞进去。再过几天，又会继续往里面补新的要求。最后它不再像一段 Prompt，更像一份谁都不想维护的长说明。

问题不在于规则太多，问题在于本来就该分层的规则，全被塞进了同一层。

## 一坨 Prompt 的问题，不是长，而是乱

一段 Prompt 最开始的作用，其实很简单，就是把这一次的任务讲明白。可一旦你想让它同时承担长期规范、场景限制和不同角色的职责，它就会很快失去边界。今天往里加一句，明天又补一条，表面上是更完整了，实际上是更脆弱了。

因为 Prompt 天生更适合解决“这次怎么做”，不适合承担“以后都按什么标准做”。把这两件事混在一起，结果就是规则越来越臃肿，最后谁也说不清哪些是这个项目真正的底线，哪些只是某次临时补上去的偏好。

更麻烦的是，这种写法根本不适合长期维护。每加一条新要求，都得继续往同一个 Prompt 里打补丁。旧规则会被埋住，新规则会互相打架，Writer 和 Reviewer 看到的也很容易不是同一个重点。到最后，Prompt 还在变长，但系统并没有变稳。

## AGENTS.md 真正重要的地方，是它能一直生长

AGENTS.md 真正重要的地方，不是它比 Prompt 更长，而是它跟 Prompt 根本不是同一种东西。Prompt 更像一次性的输入，目的是让模型把这一轮任务先做出来。AGENTS.md 不是，它更像这个项目自己的制度文件。它会在开工前被读取，而且可以按目录一层层继承、覆盖，离当前任务越近的规则优先级越高。这样一来，模型读到的就不再是一大坨一次性说明，而是和当前任务真正相关的那部分规则。

拿我写作的 Agent 来举例，这件事就很好理解。仓库根目录的 AGENTS.md 管的是整个项目的边界和工作流，比如哪些目录才是内容的唯一标准、提交该怎么分组、文章默认要走 Writer 和 Reviewer 的双 Agent 循环。再往下，writing-rules.md 把正文、TL;DR、标题、封面词拆开来写。然后 `.codex/agents/` 下面的 writer 和 reviewer 配置，再继续把两个角色的职责拆开：一个负责写，一个负责审。

这里最关键的变化是，规则不再只存在于某一轮对话里，而是已经长在这个项目里了。今天改一次，后面的任务会继续继承；明天再补一条，它也能继续长在原来的位置上，而不是又回到那段越来越长的 Prompt 里。更重要的是，它不是一股脑全塞给模型，而是按位置加载必要的信息。写文章的时候，更靠近文章任务的规则会优先生效；做复盘的时候，复盘那一层的规则再接着往上叠。这样规则才能越长越清楚，而不是越长越乱。

## 真正该稳定下来的，是标准

把这件事放到写作 Agent 里，就更容易看清楚了。真正需要长期稳定下来的，其实不是“这次写得顺不顺”，而是这个 Agent 到底怎么判断一篇文章算合格。

标题要什么样，正文要什么样，TL;DR 要怎么写，封面词要压到什么程度，什么话能说，什么话不能说，这些都应该是长期稳定下来的标准，而不是每次重来一遍的临时想法。只要这些标准还是散落在一段段 Prompt 里，写一篇文章就得重新补一次，系统自然会越写越乱。

## Prompt 是临时提示词，AGENTS.md 是能生长的系统

说到底，Prompt 和 AGENTS.md 干的不是同一件事。Prompt 负责把这一轮任务先推起来，AGENTS.md 负责把整个系统慢慢长出来。前者强调的是这次怎么做，后者强调的是以后一直怎么做。前者是临时输入，后者是长期结构。

这也是为什么，真正该做的不是继续把 Prompt 写得更长，而是尽快把 Prompt 里那些长期有效的东西抽出来，写进 AGENTS.md。只有这样，规则才会从一段临时提示词，慢慢变成一个能继承、能迭代、能继续生长的系统。
